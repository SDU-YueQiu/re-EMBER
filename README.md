# kEmber

`kEmber` 是一个围绕 EMBER（Exact Mesh Booleans via Efficient & Robust Local Arrangements）论文思路进行实现与验证的 C++17 仓库。当前重点在于精确整数几何、平面表示、凸多边形裁剪、局部 BSP 构建，以及后续环绕数分类流程的落地。

## 项目状态

截至 2026-04-22，本仓库处于“开发中 + 整理中”的原型阶段，不应视为已经收敛的完整布尔求解器。

### 当前工程现状

- [docs/API.md](docs/API.md) 已整理当前主要接口，适合快速查阅模块职责和调用约束。
- [docs/diagrams/](docs/diagrams/) 提供系统结构图和布尔流程图。
- 根目录 `CMakeLists.txt` 已建立 C++17 工程骨架，并预留了 `Eigen3` / `CGAL` 的可选开关。
- 默认可执行入口仍在整理中：`src/application/main.cpp` 目前主要保留历史实验代码，仓库内没有启用中的 `main` 入口。

## 目录说明

- `src/math/`：256 位整数、向量、行列式、符号判断、布尔状态辅助
- `src/geometry/`：平面几何基础、几何图元、裁剪与求交
- `src/algorithm/`：BSP、WNV 路径追踪等算法层逻辑
- `src/core/`：问题级组装与调度入口
- `src/application/`：应用层入口与历史实验代码
- `src/tests/`：当前的断言式测试代码
- `include/slimcpplib/`：仓库内使用的底层长整数实现
- `docs/`：接口文档、交接文档、架构图和流程图
- `reference/`：论文与参考材料整理结果

## 代码规范

**对外接口注释统一使用 Doxygen 风格**。

### 1. 注释规范：对外接口统一使用 Doxygen

适用范围：

- 头文件中的对外函数声明
- `struct` / `class` / `enum` / 类型别名等对外可见声明
- 调用约束复杂、失败语义不直观、几何含义容易误解的接口

基本要求：

- 优先在声明处写注释，而不是把接口说明散落到 `.cpp` 实现里
- 使用 `/** ... */` 块注释作为主格式
- 至少写清楚 `@brief`
- 有输入输出参数时写 `@param[in]`、`@param[out]` 或 `@param[in,out]`
- 有返回值时写 `@return`；枚举型返回状态更适合写 `@retval`
- 有前置条件、方向约定、退化条件、失败语义时，使用 `@pre`、`@note`、`@warning` 等标签补充
- 枚举值的行尾说明优先使用 `///<`

推荐模板：

```cpp
/**
 * @brief 计算两个多边形的交线载体。
 *
 * @param[in] target 当前目标多边形。
 * @param[in] incoming 待插入多边形。
 * @param[out] outSplitPlane 输出切分平面。
 * @param[out] outV0 输出第一个端点约束平面。
 * @param[out] outV1 输出第二个端点约束平面。
 * @return 当成功构造唯一有效交线时返回 `true`。
 * @note `outV0` 和 `outV1` 表示端点来源平面
 */
bool computePolygonIntersectionCarrier(
    const Polygon256& target,
    const Polygon256& incoming,
    Plane3i& outSplitPlane,
    Plane3i& outV0,
    Plane3i& outV1);
```

额外约束：

- 历史遗留的 `//` 接口说明在短期内允许存在，但只要修改了对应声明，就应顺手迁移为 Doxygen 风格
- 纯实现细节、局部算法步骤、一次性调试说明，可以继续使用普通 `//` 注释
- 不要为了“看起来有注释”而重复代码字面意思；注释应解释几何语义、约束、失败原因和符号约定

### 2. 语义优先于表面整洁

- 优先保证几何语义、符号约定、法向方向和退化条件写清楚，再谈命名和格式统一
- 当接口的实际语义与旧名字不一致时，应优先修正文档和注释，必要时再推动重命名
- 对“在内/在外/边界/无效/不支持”的区分要写清楚，不要把多种情况混成同一个模糊返回值说明

### 3. 头文件负责接口语义，源文件负责实现细节

- `.h` 文件负责回答“这个接口是什么、输入输出是什么、什么时候能用”
- `.cpp` 文件负责回答“这个接口怎么实现、为什么这样实现”
- 如果一个行为依赖平面朝向、齐次坐标符号、唯一交点等前提，这些前提必须写在声明附近

### 4. 尽量保持精确几何语义

- 当前核心数据路径建立在精确整数与行列式判定之上
- 如果某个函数只能在凸多边形、唯一交点或非共面条件下使用，应在接口注释里明确写出

### 5. 修改边界

- `include/slimcpplib/` 视为底层依赖代码，除非确有必要，否则不要顺手重排或重写风格
- 文档整理优先覆盖当前项目代码，不把第三方实现拉入统一格式化范围
